# Usage Limits — 実行上限到達時のフォールバック運用

> Claude Code本体やモデル利用上限で定例ジョブが止まることはある。  
> その時に重要なのは「自動実行に固執しない」こと。必要最小限の成果物を手動で埋めて、運用を止めない。

## 症状

- 定例ジョブが `code 1` で停止する
- Claude Codeの呼び出しが上限到達で失敗する
- HEARTBEATの一部タスクだけ未完了になる

## 原則

1. **停止理由を先に切り分ける**
   - 認証切れなのか
   - バージョン不具合なのか
   - 単純な利用上限なのか
2. **重要成果物を優先して埋める**
   - 毎日必要なサマリ
   - 明日の意思決定に必要なメモ
   - 外部公開や予約に必要な最低限の出力
3. **自動化の完全復旧を待たない**
   - その場で軽量な手動フォールバックを実行する
4. **何を手動で埋めたか記録する**
   - 後で自動化を修復するため、代替対応の痕跡を残す

## 推奨フロー

### 1. まず原因を分類

- OAuth切れ → [[Auth Expiry]]
- バージョン起因の異常 → [[Version Bugs]]
- MCP/ブラウザ経由で落ちる → [[MCP Crash]]
- それ以外で、単に実行枠・利用上限に詰まっている → このページの対応

### 2. タスクを「今夜必要な成果物」に縮約

例:
- `/daily-sync` が止まった → Daily note の重要事項だけ手動整理
- `obsidian-connector` が止まった → 接続メモや関連リンクだけ手動で残す
- 投稿生成が壊れた → ルール準拠の最小ドラフトだけ作って予約/保存する

**原則:** 自動タスクを丸ごと再現しようとしない。  
意思決定と翌日の行動に必要な最小出力だけ先に作る。

### 3. 生成と実行をさらに分離

利用上限時は、重いエージェント実行を避けて以下に寄せる。

- 人間 or 軽量手順でドラフトを作る
- APIや既存スクリプトで実行だけ行う
- 成果物を先にファイルへ保存する

これは [[Two-Phase Execution]] の緊急時版と考えるとよい。

### 4. 復旧後に恒久対策を残す

- どの定例が止まったか
- 何を手動で代替したか
- 次回から何を軽量化/分割すべきか

## 実運用で有効だったパターン

### パターンA: 定例ノート系は「要点のみ手動化」
- 自動実行が止まっても、その日の優先事項・未完了事項・翌日の重要イベントだけは手で残す
- これで翌朝の判断品質はほぼ維持できる

### パターンB: 生成スクリプトが壊れても予約だけ進める
- 出力品質ルールが既に決まっているなら、ドラフトを手で整えてAPI/既存導線で予約する
- 「生成が壊れた = 投稿全停止」にしない

### パターンC: 接続メモは薄くても空欄にしない
- connector系タスクは完全自動化にこだわらず、最低限の関連づけを手動で残す
- 後続の検索性と継続性を守れる

## アンチパターン

- 上限到達後も同じ重いコマンドを繰り返し再実行する
- 自動タスクが失敗したので成果物をゼロのまま放置する
- 手動フォールバックした事実を記録しない
- 「今日は壊れていた」で済ませて再発防止を書かない
