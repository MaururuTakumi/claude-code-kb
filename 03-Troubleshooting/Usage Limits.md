# Usage Limits — 実行上限到達時のフォールバック運用

> Claude Code本体やモデル利用上限で定例ジョブが止まることはある。  
> その時に重要なのは「自動実行に固執しない」こと。必要最小限の成果物を手動で埋めて、運用を止めない。

## 症状

- 定例ジョブが `code 1` で停止する
- Claude Codeの呼び出しが上限到達で失敗する
- HEARTBEATの一部タスクだけ未完了になる

## 原則

1. **停止理由を先に切り分ける**
   - 認証切れなのか
   - バージョン不具合なのか
   - 単純な利用上限なのか
2. **重要成果物を優先して埋める**
   - 毎日必要なサマリ
   - 明日の意思決定に必要なメモ
   - 外部公開や予約に必要な最低限の出力
3. **自動化の完全復旧を待たない**
   - その場で軽量な手動フォールバックを実行する
4. **何を手動で埋めたか記録する**
   - 後で自動化を修復するため、代替対応の痕跡を残す

## 推奨フロー

### 1. まず原因を分類

- OAuth切れ → [[Auth Expiry]]
- バージョン起因の異常 → [[Version Bugs]]
- MCP/ブラウザ経由で落ちる → [[MCP Crash]]
- それ以外で、単に実行枠・利用上限に詰まっている → このページの対応

### 2. タスクを「今夜必要な成果物」に縮約

例:
- `/daily-sync` が止まった → Daily note の重要事項だけ手動整理
- `obsidian-connector` が止まった → 接続メモや関連リンクだけ手動で残す
- 投稿生成が壊れた → ルール準拠の最小ドラフトだけ作って予約/保存する

**原則:** 自動タスクを丸ごと再現しようとしない。  
意思決定と翌日の行動に必要な最小出力だけ先に作る。

### 3. 生成と実行をさらに分離

利用上限時は、重いエージェント実行を避けて以下に寄せる。

- 人間 or 軽量手順でドラフトを作る
- APIや既存スクリプトで実行だけ行う
- 成果物を先にファイルへ保存する

これは [[Two-Phase Execution]] の緊急時版と考えるとよい。

### 4. 復旧後に恒久対策を残す

- どの定例が止まったか
- 何を手動で代替したか
- 次回から何を軽量化/分割すべきか

## 実運用で有効だったパターン

- 2026-04-22: xAI credits / spending limit 到達で `x-viral-ai-search` は本文 `（取得データなし）` になったが、raw JSON にエラー本文が残った。朝報や下書きでは raw を証跡として扱えば、検索系タスクを全停止にせず継続できる。
- 2026-04-22: watchdog が `hb-main-analytics` / `hb-main-recon` / `hb-main-post` の欠落を同日朝に補完し、AM Sweep と投稿準備を復旧できた。定例の取りこぼしは「翌サイクルで即補完」を標準動作にする。
- 2026-04-22: AGA投稿は生成失敗時でも、手動文面作成 + Typefully publish で公開まで復旧できた。生成失敗と公開失敗は分けて扱う。
- 2026-04-23: Claude organization access エラーと xAI 429 / no-data が同日に重なっても、検索系は raw JSON や既存スキャン結果を証跡として残し、生成系は OpenClaw 側手動ドラフトへ縮約すれば主要ラインを維持できた。
- 2026-04-23: 22時台の同期系（daily-sync, Omi, Obsidian, reality-sync）は Claude 抽出失敗を前提に session logs と成果物確認で救済した方が安全。夜間同期系は「Claude落ち前提の救済フロー」を標準装備する。
- 2026-04-23: AGA夜運用は competitor scan → quote RT候補 → 手動ドラフト → SEO記事化まで連結できた。利用上限時は重い再実行を繰り返すより、既存リサーチを横展開して複数成果物へ再利用する方が強い。
- 2026-04-24: `x-reply-check` は xAI credits / monthly limit 到達でも、API失敗をそのまま全停止扱いせず、ログイン済みブラウザの mentions を見に行く fallback を持たせると返信要否の判定までは維持できる。
- 2026-04-24: SNS運用では「publish 成功」と「engage 失敗」を別ステータスで記録する。公開後の後続処理だけが落ちた時に、投稿自体まで blocked 扱いすると watchdog と日次レビューが歪む。
- 2026-04-26: xAI credits / monthly spending limit 到達が複数タスクに波及しても、Chrome mentions fallback・X API・Web/公式ソース・既存リサーチの手動キュレーションを組み合わせれば、返信確認・投稿準備・週次レビューは維持できる。
- 2026-04-26: 生成スクリプトが Claude organization access エラーで落ちても、方針とフォーマットが固定済みなら「手動ドラフト作成 → 既存 publish スクリプト/APIで実行」に縮約して公開枠を守れる。生成失敗と配信失敗は必ず分けて記録する。
- 2026-04-26: browser automation が `Failed to start session server` で落ちる日は、live dashboard / engage は blocked として切り分け、既存 analytics・sales log・published artifacts を使ったファイルベースレビューに切り替える。レビュー全体を止めない。
- 2026-04-26: watchdog recovery は「state 未記録」と「artifact 既存」を照合し、artifact があるものは state 補完、artifact がないものだけ最小成果物を再生成する。全タスクを再実行しない方が速く安全。

### パターンA: 定例ノート系は「要点のみ手動化」
- 自動実行が止まっても、その日の優先事項・未完了事項・翌日の重要イベントだけは手で残す
- これで翌朝の判断品質はほぼ維持できる

### パターンB: 生成スクリプトが壊れても予約だけ進める
- 出力品質ルールが既に決まっているなら、ドラフトを手で整えてAPI/既存導線で予約する
- 「生成が壊れた = 投稿全停止」にしない

### パターンC: 接続メモは薄くても空欄にしない
- connector系タスクは完全自動化にこだわらず、最低限の関連づけを手動で残す
- 後続の検索性と継続性を守れる

### パターンD: no-data / partial を成果物として確定する
- 検索APIや外部LLMが 429 / credits不足で空振りしても、`（取得データなし）` と raw error を保存して次工程へ渡す
- 「未取得」と「未実行」を分けるだけで、watchdog や翌朝補完がかなり安定する

### パターンE: 既存リサーチを横展開して夜ラインを守る
- 生成モデルが落ちても、その日の competitor scan や候補メモから X投稿, quote RT, SEO記事へ再構成する
- 新規検索を諦めても、翻訳・再編集だけで十分に回る日は多い

## アンチパターン

- 上限到達後も同じ重いコマンドを繰り返し再実行する
- 自動タスクが失敗したので成果物をゼロのまま放置する
- 手動フォールバックした事実を記録しない
- 「今日は壊れていた」で済ませて再発防止を書かない
