# Harness Design — AIをどう動かすかを設計する

> **ハーネス設計とは、AIに何をさせるかではなく、AIをどう動かすかを設計する技術である。**

Claude Code運用では、モデル単体の性能よりも、
**役割分離・評価・差し戻し・実行順序・中間成果物**の設計で品質が決まる。

関連: [[Two-Phase Execution]] [[Model Selection]] [[Self-Improving Loop]]

## なぜ必要か

単一エージェントで長い複雑タスクを丸ごと処理させると、後半で品質が崩れやすい。
ハーネス設計は、この崩壊を**構造で防ぐ**ためのパターン。

### ソロ実行が崩れる主因

#### 1. Context Anxiety
- 長時間タスクで後半ほど要件や制約が抜ける
- 「完成した」と言うが未完成になりやすい
- 文脈全保持を前提にすると、終盤で一貫性が落ちる

#### 2. Self-Evaluation Bias
- 作った本人に評価させると甘くなる
- AIは自分の生成物を高く見積もりやすい
- **生成役と評価役を分けない限り改善しにくい**

## 中核パターン: Planner → Generator → Evaluator

### Planner
- ユーザーの短い依頼を仕様書に展開する
- 要件・制約・実行順序を明確化する
- 完成条件を定義する

### Generator
- 計画に基づいて実装・生成・実行する
- スプリント単位で成果物を出す
- 完了したら Evaluator に渡す

### Evaluator
- 実際に触って壊す
- バグ・抜け漏れ・品質不足を検出する
- スコアが閾値未満なら差し戻す

**重要:** Evaluator は感想を言う役ではない。**具体的な失敗を見つける役**。

### Evaluator のチューニングループ

Evaluator は最初から優秀ではない。以下のループで精度を上げる:

1. **ログを読む**: Evaluator の採点結果を確認する
2. **ズレを見つける**: 自分の判断と Evaluator の判断が食い違う箇所を特定
3. **プロンプトを修正**: ズレの原因（甘すぎる/見落とす/表層的すぎる）を解消
4. **再実行して検証**: 修正後のプロンプトで再度評価させて改善を確認
5. **Few-shot でキャリブレーション**: スコア例（「このアウトプットは7点、理由は…」）を数件プロンプトに含めると、Evaluator の判断が安定する

**出典:** Anthropicの実験では、Evaluatorが正当な問題を見つけた後「大した問題ではない」と自分を説得して承認するケースが頻発。数ラウンドのチューニングで解消。

## Sprint Contract（事前合意プロセス）

実装前に、Generator と Evaluator の間で
**「何をもって完成とするか」** を交渉して合意する。

### 交渉プロセス
1. **Generator が提案**: 「このスプリントで何を作るか」「どうやって検証するか」を提示
2. **Evaluator がレビュー**: 「正しいものを作ろうとしているか」を確認。不足があれば差し戻す
3. **合意するまで繰り返す**: 双方が納得してから実装開始

### 粒度の目安
Anthropicの実験では、1スプリントに**27個の検証基準**を設けた例がある。
粒度は「Evaluatorが自動テストできるレベル」まで落とす。

### 例
- Generator: 「矩形塗りつぶしツールを実装する。ドラッグ範囲を塗れるようにする」
- Evaluator: 「検証基準は以下」
  - ドラッグ範囲全体が塗りつぶされる
  - Undo が機能する
  - 開始点/終了点だけでなく中間も反映される

**教訓:** 実装前に Definition of Done を固定すると、差し戻しコストを減らせる。
仕様が高レベルなほど、Sprint Contract の交渉で実装の方向を固めることが重要。

## Rubric-based Evaluation

評価は「良い/悪い」ではなく、**採点基準**で行う。

### 原則
- 主観をそのままレビューに使わない
- 「この基準を満たしているか」に変換する
- 評価軸には**重み付け**をする

### なぜ重み付けが必要か
AIは Craft や Functionality のような基礎点は自然に取りやすい。
一方で、人間が本当に欲しい差分は Design Quality や Originality のような軸に出る。

**全部を均等評価しないこと。**
差がつく軸を重く採点する。

## ファイルベース通信（Inter-Agent Communication via Files）

エージェント間の状態受け渡しは**ファイル経由**で行う。

```
Agent A → /tmp/checkpoint/step1.md に書く
Agent B → /tmp/checkpoint/step1.md を読んで作業 → step2.md に書く
Agent C → step2.md を読んで評価 → eval.md に書く
```

### なぜファイルか
- プロンプト内で口頭で渡すと、コンテキスト圧迫で情報が落ちる
- ファイルなら途中で死んでも**中間成果物が残る**
- 人間が読んで監査できる（透明性）
- バージョン管理できる

### 実装パターン
- `/tmp/checkpoint/[task]-step[N].md` に統一
- 各ステップ開始時にクリーンアップ
- 最終成果物だけ永続ディレクトリ（memory/, drafts/）に移動

**出典:** Anthropicのハーネスでは「一方のエージェントがファイルを書き、もう一方がそのファイルを読んで応答する」方式を採用。

## ハーネスは速さや安さのためではない

ハーネス設計は、安く速く済ませるための技法ではない。
むしろ、
- 時間は増える
- コストも増える

その代わり、**単一エージェントでは届かない品質上限**に到達できる。

### 比較の基準
比較すべきなのは:
- 実行時間
- トークンコスト

ではなく:
- **目的達成率**
- **完成品質**
- **再現性**

**安くても完成していなければ失敗。**

## 新モデルが出たらハーネスを見直す

ハーネスは固定化された正解ではない。
モデルが改善されたら、最適な分割粒度・リセット頻度・役割構成も変わる。

### 教訓
- 古い制約を惰性で残さない
- 新モデルで不要になった工程は削る
- ただし、役割分離そのものの価値は消えない

**モデル進化 = ハーネス再設計のタイミング**

### ハーネス簡素化テスト

ハーネスが複雑になりすぎたら、以下の手順で削ぎ落とす:

1. **コンポーネントを1つずつ外す**（一気に複数外さない）
2. **同じタスクを実行して品質を比較する**
3. **品質が下がらなければ、そのコンポーネントは不要**
4. **品質が下がれば戻す**

**原則:** "find the simplest solution possible, and only increase complexity when needed"
（Anthropic "[Building Effective Agents](https://www.anthropic.com/research/building-effective-agents)"）

### Evaluator の費用対効果

Evaluator は**常に必要ではない**。

- タスクがモデルの能力範囲内 → Evaluator不要（コストの無駄）
- タスクがモデルの能力の**境界**にある → Evaluatorが最も効果を発揮
- 能力の境界はモデル更新のたびに移動する

**判断基準:** そのタスクを Evaluator なしで5回実行して、3回以上合格品質が出るなら Evaluator は省略できる。

## いつ使うか

次の条件を満たすなら、ハーネスを検討する:
- 長時間・多工程タスク
- 複数ファイル変更
- 外部公開物（投稿・提案・営業文・記事）
- お金や信頼に直結するアウトプット
- 「生成したら終わり」ではなく、品質担保が必要な作業

## 今日から使える5原則

1. **Generator と Evaluator を分離する**
2. **「何を作るか」を先に合意する**
3. **主観をルーブリック化する**
4. **ハーネスはできるだけシンプルに保つ**
5. **新モデルが出たらハーネスを見直す**

## Claude Code運用への翻訳

### 最小構成
- Opus: Planner
- Sonnet: Generator
- Opus もしくは別セッション: Evaluator

### さらに強化する場合
- Codex: 分析・スキャン・比較の補助
- Playwright / browser-use / browser tool: 実機検証
- スコアカード: rubric をファイル化して差し戻し条件を固定

## 具体適用例

### 外部公開コンテンツ
- Planner: 投稿の狙い・制約・成功条件を定義
- Generator: 投稿案を生成
- Evaluator: 機密情報、事実性、読者価値、トーンを採点

### アフィリエイト案件
- Planner: 訴求角度、禁止表現、比較軸を定義
- Generator: LP案・投稿案・訴求文を生成
- Evaluator: 景表法/誇張/事実誤認/売上導線を採点

### ブラウザ自動化
- Planner: フローを仕様化
- Generator: 実装
- Evaluator: 実機で壊してバグを報告

## まとめ

> **AIの限界は、モデル性能の限界より先に、運用構造の限界として現れる。**

Claude Codeを本番運用に載せるなら、
プロンプトの工夫だけでなく、**役割分離と評価設計**まで含めて設計すること。
