# Self-Improving Loop — 自己改善ループの設計

> Claude Codeを使うほど、Claude Codeの使い方が良くなる。
> これを偶然ではなく、仕組みとして組み込む。

## なぜ自己改善ループが必要か

CLAUDE.md やスキルは**書いた時点の知識**で作られている。
しかし実際に動かすと:
- 「この手順書だとSonnetが迷う」
- 「このタスクはOpusじゃなくてSonnetで十分だった」
- 「このMCPパターンは3回連続で失敗してる」

こういう発見は**実行してみないとわからない**。
だから「実行 → 記録 → 分析 → 改善」のループを回す。

## ループの構造

```
┌─────────────────────────────────┐
│ 1. 実行                         │
│    Claude Codeがタスクを実行      │
│    CLAUDE.md + Skills に従って動く │
└────────────┬────────────────────┘
             ↓
┌─────────────────────────────────┐
│ 2. 記録                         │
│    結果を構造的に記録する          │
│    - 成功/失敗                    │
│    - 使用モデル + ターン数         │
│    - 品質スコア（該当する場合）    │
│    - 気づき（1行メモ）            │
└────────────┬────────────────────┘
             ↓
┌─────────────────────────────────┐
│ 3. 分析                         │
│    パターンを抽出する             │
│    - 同じ失敗が繰り返されてないか  │
│    - モデル選択は最適だったか      │
│    - 手順書の品質は十分だったか    │
│    - もっと効率的なやり方はないか  │
└────────────┬────────────────────┘
             ↓
┌─────────────────────────────────┐
│ 4. 改善                         │
│    設定・スキル・KBを更新する      │
│    - CLAUDE.md のタスク分類表を修正│
│    - Skills の手順を改善          │
│    - KBに新パターン/教訓を追加    │
│    → git push → 全環境に反映     │
└────────────┬────────────────────┘
             ↓
             🔁 次の実行はより良くなる
```

## 記録のフォーマット

タスク完了時に、ログファイルに以下を追記する:

```markdown
### タスク名 (モデル, ターン数)
- 結果: ✅ 成功 / ❌ 失敗 / ⚠️ 部分成功
- 品質: [スコア or 主観評価]
- 気づき: [次回改善できること]
- 改善アクション: [具体的に何を変えるか]（なければ「なし」）
```

## 分析の観点

### モデル選択の最適化
- Opusで実行したが、Sonnetで十分だったタスクはないか
- Sonnetで実行して失敗したが、Opusなら成功したタスクはないか
- → [[Model Selection]] のタスク分類表を更新

### 手順書の品質
- Sonnetが迷って停止したポイントはどこか
- 手順書のどのステップが曖昧だったか
- → [[Two-Phase Execution]] の例を改善

### パターンの発見
- 3回以上成功したやり方 → Patterns/ に昇格
- 3回以上失敗したやり方 → Troubleshooting/ に記録
- 新しいツール/APIの活用法 → Patterns/ に追加

## 改善のルール

- **小さく頻繁に改善する**: 1日1改善でも、1ヶ月で30改善になる
- **根拠を残す**: なぜ変えたかを記録する（Decisions/ に残してもよい）
- **改悪を恐れない**: git で戻せる。試して悪かったら戻す
- **改善は即反映**: 「あとでやる」は忘れる。気づいた時にやる

## このループ自体の改善

このループの回し方自体も改善対象:
- 記録が面倒すぎて続かない → フォーマットを簡略化
- 分析に時間がかかりすぎる → 自動集計スクリプトを作る
- 改善が反映されてない → 週次チェックで検証
