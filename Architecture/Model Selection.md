# Model Selection — Opus / Sonnet / Codex の使い分け

> Claude Codeのモデル使い分けは「タスクを丸ごと振り分ける」のではなく、  
> **タスクの中の「思考」と「実行」を分離して、適切なモデルに割り当てる**。

## 設計思想

```
❌ タスク丸ごとSonnet → 判断に迷って品質低下
❌ タスク丸ごとOpus → 週間リミット即死
✅ Opusが考えて手順書を書く → Sonnetが手順書通りに実行
```

関連: [[Two-Phase Execution]]

## モデル特性

| モデル | 強み | 弱み | 使いどころ |
|--------|------|------|-----------|
| **Opus** | 複雑な判断・設計・戦略・品質 | 高コスト・リミットあり | 思考・設計・品質判定 |
| **Sonnet** | 高速・低コスト・実行力 | 複雑な判断は苦手 | 定型実行・手順書通りの作業 |
| **Codex (gpt-5.4)** | トークン無制限・コード生成 | 書き込み制限（サンドボックス） | 分析・データ処理・並列スキャン |

## 判断フロー

```
タスクを受け取る
  ↓
Q1: このタスクに「判断」が含まれるか？
  → No → Sonnet単独（パターンB）
  → Yes ↓

Q2: 判断と実行を分離できるか？
  → Yes → 2段階実行（パターンA）
         Opus: 判断して手順書を書く（5-15ターン）
         Sonnet: 手順書通りに実行（15-30ターン）
  → No → Opus単独（パターンC）

Q3: コード生成・データ処理が大量にあるか？
  → Yes → Codex に分析を委任
```

## パターン詳細

### パターンA: 2段階実行
```bash
# Phase 1: Opusが設計（短い）
claude --max-turns 10 -p "Phase1: 分析方針を決めて手順書を/tmp/plan.mdに出力"

# Phase 2: Sonnetが実行（長い）
claude --model sonnet --max-turns 30 -p "Phase2: /tmp/plan.mdの指示通りに実行"
```

**Opusの手順書の品質基準:**
- Sonnetが読んで**1秒も迷わない**具体性
- 各ステップの完了条件が明確
- 判断が必要なポイントには「ここで止まってOpusに戻す」と明記

### パターンB: Sonnet単独
```bash
claude --model sonnet --max-turns 20 -p "/worklog"
```

適用条件:
- 手順が完全に固定されている
- 判断ポイントがない
- ファイル操作・ログ記録・ステータス確認のみ

### パターンC: Opus単独
```bash
claude --max-turns 50 -p "/pdca-review"
```

適用条件:
- 判断の密度が高く、分離するとオーバーヘッドの方が大きい
- Chrome MCP操作 + 判断が交互に発生する
- 戦略レビュー・週次PDCA

## Codexとの併用

```bash
# Opus が方針決定 → Codex が並列分析 → Opus が集約判断
codex exec "ファイルAを分析してJSONで出力" &
codex exec "ファイルBを分析してJSONで出力" &
wait
# → 結果をOpusが読んで最終判断
```

参照: [[Agent Teams]]

## ⚠️ リミット管理

- Opus の週間リミットは有限。[[Token Budget]] で消費を追跡する
- リミットに近づいたら全タスクをSonnet + Codex に切り替える
- リミット到達 → 即座にタクミに報告（勝手に止めない・隠さない）
